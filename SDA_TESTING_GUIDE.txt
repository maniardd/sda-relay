═══════════════════════════════════════════════════════════════════════════════
  TEST CASE 1 — ENDPOINT GETS DHCP IP (proves DHCP relay over fabric)
═══════════════════════════════════════════════════════════════════════════════
Setup:
  1. Plug laptop NIC → Edge switch port Gi1/0/3
  2. Laptop set to DHCP (default Windows / macOS / Linux)
  3. Wait 30 seconds

Expected on laptop (Windows: `ipconfig /all`):
  IPv4 Address ........ 10.30.100.11 (or .12, .13)
  Subnet Mask ......... 255.255.255.0
  Default Gateway ..... 10.30.100.1
  DHCP Server ......... 10.255.255.1
  DNS Servers ......... 8.8.8.8

Verification on Border:
  show ip dhcp binding
  show ip dhcp pool CORP_DATA_POOL

Verification on Edge:
  show interfaces Vlan100              ← line protocol must be UP
  show interfaces Gi1/0/3 status       ← connected, vlan 100
  show device-tracking database        ← laptop MAC visible
  show mac address-table vlan 100      ← laptop MAC visible

PASS: Laptop has 10.30.100.x address.
FAIL: 169.254.x.x → see "Troubleshooting DHCP" section.

═══════════════════════════════════════════════════════════════════════════════
  TEST CASE 3 — EAST-WEST: PING ACROSS THE FABRIC (proves LISP + VXLAN)
═══════════════════════════════════════════════════════════════════════════════
This is THE test. Forces edge to do LISP map-request → border map-reply →
edge VXLAN-encaps the packet → fabric link → border decaps → reply path.

From laptop:
  ping 10.30.100.254

Expected: First 1-2 pings may drop (LISP cache miss → map-request), then replies.
PASS: Replies received → SDA datapath PROVEN END-TO-END.

Verification on Edge (after the ping):
  show lisp instance-id 4099 ipv4 map-cache
      → must show entry: 10.30.100.254/32  via Locator 10.255.255.1
  show lisp instance-id 4099 ipv4 map-cache 10.30.100.254
      → detailed view: state complete, encap vxlan
  show platform software fed switch active ip route vrf CORP_VN
  show ip cef vrf CORP_VN 10.30.100.254
      → next-hop should resolve to LISP0 / locator 10.255.255.1

Verification on Border:
  show lisp instance-id 4099 ipv4 server
      → site site_uci registered, EID 10.30.100.254/32 from 10.255.255.2
  show lisp instance-id 4099 ipv4 database
      → 10.30.100.254/32 locally configured
  show ip arp vrf CORP_VN
      → laptop's IP 10.30.100.x learned via LISP/VXLAN

═══════════════════════════════════════════════════════════════════════════════
  TROUBLESHOOTING — DHCP fails (laptop gets 169.254.x.x)
═══════════════════════════════════════════════════════════════════════════════
Step 1 — confirm laptop is sending DISCOVER
  Edge: debug ip dhcp server packet detail
  (you should see DISCOVER from 0.0.0.0; remember to "undebug all" after)

Step 2 — confirm helper-address present
  Edge: show running-config interface Vlan100 | inc helper
  Expected: "ip helper-address vrf CORP_VN 10.255.255.1"

Step 3 — confirm edge can reach border in VRF
  Edge: ping vrf CORP_VN 10.255.255.1 source Loopback0
  → if this fails, LISP map-cache for 10.255.255.1 is missing.
     Check:  show lisp instance-id 4099 ipv4 map-cache
     Check:  show lisp session     (must be Up)

Step 4 — confirm DHCP pool on border serves the VRF
  Border: show ip dhcp pool CORP_DATA_POOL
  Border: show running-config | section ip dhcp pool

Step 5 — confirm border DHCP relay accepts cross-VRF
  Border: show running-config | inc dhcp
  Expected: "ip dhcp relay information trust-all"

═══════════════════════════════════════════════════════════════════════════════
  TROUBLESHOOTING — LISP session DOWN
═══════════════════════════════════════════════════════════════════════════════
On Border:
  show lisp session     ← shows local sessions
If peer 10.255.255.2 is "Down":
  1. ping 10.255.255.2 source Loopback0  (must succeed; ISIS underlay)
  2. show ip route 10.255.255.2          (must be /32 via ISIS)
  3. show running-config | section router lisp
     → confirm "site site_uci" exists with correct authentication-key
  4. On Edge: show running-config | section router lisp
     → confirm "etr map-server 10.255.255.1 key CiscoSDA123"
     → key MUST match border's site authentication-key
  5. clear lisp session on both sides; re-check after 30s

═══════════════════════════════════════════════════════════════════════════════
  TROUBLESHOOTING — Ping 10.30.100.254 fails
═══════════════════════════════════════════════════════════════════════════════
1. From Border: ping vrf CORP_VN 10.30.100.254 source Loopback4099
   → must succeed (proves the loopback exists and VRF routing local)

2. From Border: ping vrf CORP_VN 10.30.100.<laptop-ip>
   → if fails, border has no LISP entry for laptop. Check:
     show lisp instance-id 4099 ipv4 server
     → laptop /32 should be registered from 10.255.255.2
   If missing, on Edge: show device-tracking database
     → laptop must be tracked first (IPDT triggers LISP register)

3. From Edge: show lisp instance-id 4099 ipv4 map-cache 10.30.100.254
   → if "no entry", LISP control-plane is broken (see Troubleshooting LISP)

4. Verify VXLAN encap path: show ip cef vrf CORP_VN 10.30.100.254 detail
   → must show Locator 10.255.255.1, encap vxlan
