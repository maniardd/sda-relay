A. RELAY SERVER (VM) - FULL SPECIFICATIONS
================================================================

PURPOSE:
  The relay server bridges Meraki Cloud (internet) to the
  on-prem switches (private IPs). Meraki sends HTTPS webhooks
  to this server, which translates them into RESTCONF API
  calls to configure the Catalyst switches.

---------- HARDWARE / VM SPECS ----------

  Resource        Minimum (Lab)      Recommended (Prod)
  -----------     --------------     ------------------
  vCPU            2 cores            4 cores
  RAM             4 GB               8 GB
  Disk            40 GB (SSD)        80 GB (SSD)
  NIC             1x 1GbE           2x 1GbE

  NOTE: A single NIC is fine for the POC. It will carry
  both management traffic AND RESTCONF traffic to switches.

---------- OPERATING SYSTEM ----------

  OS:             Ubuntu 22.04 LTS Server (64-bit)
  Download:       https://releases.ubuntu.com/22.04/
  ISO File:       ubuntu-22.04.4-live-server-amd64.iso (~1.8 GB)
  Kernel:         5.15 LTS (default with 22.04)

  WHY Ubuntu 22.04:
    - Free (no license)
    - Python 3.10 included
    - LTS support until April 2027
    - ~2 GB RAM base usage (leaves room for the app)

---------- VM PLATFORM OPTIONS ----------

  Option 1: ESXi / vSphere  (preferred if available)
  Option 2: Hyper-V on Windows Server
  Option 3: VirtualBox on any PC (quick for lab)
  Option 4: Bare-metal (Intel NUC or similar)

---------- NETWORKING ON THE VM ----------

  VM Network Adapter Mode: BRIDGED (not NAT)
    -> Must bridge to the physical NIC on the same VLAN/subnet
       that can reach 192.168.128.0/24 (switch management)

  Static IP Configuration on the VM:
    IP Address:   192.168.128.10
    Subnet Mask:  255.255.255.0  (/24)
    Gateway:      192.168.128.1  (MX or default GW)
    DNS:          208.67.222.222, 8.8.8.8
    NTP:          Same NTP server used by switches

  All devices are on the SAME subnet: 192.168.128.0/24

---------- SOFTWARE TO INSTALL ON VM ----------

  After Ubuntu installation, run:

    sudo apt update && sudo apt upgrade -y
    sudo apt install -y python3 python3-pip python3-venv git curl net-tools

  Then:
    mkdir -p /opt/sda-relay && cd /opt/sda-relay
    python3 -m venv venv
    source venv/bin/activate
    pip install flask requests pyyaml python-dotenv gunicorn

  ngrok (HTTPS tunnel):
    curl -sSL https://ngrok-agent.s3.amazonaws.com/ngrok-v3-stable-linux-amd64.tgz \
      | sudo tar xz -C /usr/local/bin
    ngrok config add-authtoken <TOKEN_FROM_ngrok_dashboard>

---------- VM USER ACCOUNT ----------

  Username:    sdaadmin
  Hostname:    sda-relay-server
  SSH:         Enable OpenSSH server during install
  Privilege:   Full sudo access

B. SWITCH CONFIGURATION REQUIREMENTS
================================================================

These are the MINIMUM configs needed on BOTH switches so they can:
  1. Be claimed into Meraki Dashboard (Cloud Monitoring)
  2. Accept RESTCONF API calls from the relay server

  *** IMPORTANT: DO NOT configure SDA-specific items manually! ***
  The relay server will push ALL SDA configuration automatically:
    - Hostnames, Loopback0, fabric link IPs
    - ISIS, LISP, VXLAN, PIM, BGP
    - VN overlays (CORP_VN, GUEST_VN)
  You only need: management IP, RESTCONF, AAA, DNS, NTP, Meraki connect.

---------- B1. COMMON CONFIG (BOTH SWITCHES) ----------

! === RESTCONF (Required for relay server) ===
conf t
restconf
ip http server
ip http secure-server
ip http authentication local

! === AAA (Required for both Meraki + RESTCONF) ===
aaa new-model
aaa authentication login default local
aaa authorization exec default local

! === Local User with Privilege 15 (Required) ===
! This user is used by: (a) RESTCONF from relay, (b) Meraki onboarding
username admin privilege 15 secret <STRONG_PASSWORD>

! === IP Routing (Required for Meraki tunnel) ===
ip routing

! === DNS Resolution (Required for Meraki tunnel) ===
ip name-server 208.67.222.222
ip domain lookup
ip domain name sjc23.cisco.com

! === NTP (Required - clock must be correct for TLS) ===
ntp server 10.40.0.1
! (Use your lab NTP server. If none, use: ntp server pool.ntp.org)

! === System MTU ===
system mtu 9100

! === Meraki Cloud Registration ===
! This initiates the tunnel to Meraki Dashboard
service meraki connect
end

! === VERIFY MERAKI TUNNEL ===
show meraki connect
! Expected output:
!   Meraki Tunnel status: Up
!   Cloud ID: XXXX-XXXX-XXXX    <-- RECORD THIS! Needed to claim in Dashboard
!
! If tunnel is DOWN, check:
!   - DNS resolution (ping google.com)
!   - NTP (show clock -- must be current date/time)
!   - Internet reachability (ping 8.8.8.8)
!   - Firewall rules (port 443 outbound)

---------- IMPORTANT CABLING NOTES ----------

  * Cable #4 (Fabric Link) MUST be a DIRECT connection between the
    two switches. No switch in between. This is the SDA underlay.

  * For Cable #4, you need a compatible SFP28/SFP+ transceiver:
    - C9500 side: 25G SFP28 port -> use SFP-25G-AOC or SFP-10/25G-CSR-S
    - C9300X side: 10G SFP+ port in the network module slot (1/1/x)
    - If mixing speeds, port auto-negotiates to the lower speed (10G)
    - A 10G SFP+ DAC cable works on both sides if 25G is not required

  * For management (Cables #2 and #3):
    - Use a FRONT-PANEL port configured as access on VLAN 128
    - Do NOT use GigE0/0 (the OOB management port on the back)

  *** CRITICAL ***
      Meraki "service meraki connect" requires the uplink to be
      via a FRONT-PANEL port, NOT the dedicated management port
      (GigE0/0). The Meraki tunnel will NOT form over GigE0/0.
      You CAN still use GigE0/0 for local console/SSH if desired.

E. INTERNET & CLOUD REQUIREMENTS
================================================================

---------- DOES THE VM NEED INTERNET? ----------

  YES. Outbound HTTPS (port 443) to:
    1. *.ngrok.com    - HTTPS tunnel service
    2. api.meraki.com - Dashboard API
    3. Ubuntu repos   - package updates (apt)

---------- DO THE SWITCHES NEED INTERNET? ----------

  YES. Outbound HTTPS (port 443) to:
    1. *.meraki.com   - Dashboard + Cloud tunnel
    2. DNS server     - To resolve Meraki hostnames
    3. NTP server     - Clock sync (TLS needs correct time)

---------- MERAKI-SPECIFIC FIREWALL RULES ----------

  If there's a firewall, allow these domains OUTBOUND on port 443:

    *.meraki.com
    *.networkmeraki.com
    dashboard.meraki.com
    api.meraki.com

  Full list: https://documentation.meraki.com/General_Administration/
             Other_Topics/Upstream_Firewall_Rules_for_Cloud_Connectivity

---------- DOES ANYTHING NEED INBOUND INTERNET ACCESS? ----------

  NO. Everything is OUTBOUND only:
    - ngrok: outbound tunnel (VM --> ngrok servers)
    - Meraki: outbound tunnel (switch --> Meraki cloud)
    - No inbound firewall rules needed
    - No public IP needed
    - No port forwarding needed
